# Cloudflare Quick Tunnel protocol compatibility

This implementation is pinned to `cloudflared 2026.7.2`, commit
`8679787525edc8575b2948a7c4a50b6292c6d426`. The Go source is a behavioral
reference and is not compiled or copied into the product.

The Quick Tunnel API request uses the matching `cloudflared/2026.7.2` user
agent and cloudflared's 15-second per-request timeout. Transient service 5xx
responses are retried within the Supervisor's bounded startup deadline.

## Edge transport contract

- Discovery: `_v2-origintunneld._tcp.argotunnel.com` SRV, honoring priority and weight.
- Network: direct outbound TCP to the discovered port, normally 7844.
- TLS: client SNI `h2.cftunnel.com`, native roots plus the pinned Cloudflare roots, and an empty ALPN list.
- HTTP/2 roles: Dinotty is the H2 server; the edge is the H2 client and opens request streams.
- Limits: 128 concurrent streams per edge connection, 64 KiB header list, 1 MiB stream receive window, and 4 MiB connection receive window.
- Control stream: `Cf-Cloudflared-Proxy-Connection-Upgrade: control-stream`; one per connection; `200` response without END_STREAM followed by bidirectional Cap'n Proto RPC DATA.
- Liveness: H2 PING frames are handled by the `h2` connection driver. TCP keepalive starts after 10 seconds with 5-second probes.

## Registration contract

- One connector UUID is generated per manager lifetime and reused by every HA leg and reconnect.
- Edge connection UUIDs remain per-registration values.
- `replaceExisting` is always false.
- `numPreviousAttempts` counts complete failed attempts for the leg and saturates at 255.
- Connections report the edge TCP socket's local IP.
- The H2-only feature list is exactly `serialized_headers`.
- Connection zero sends best-effort `UpdateLocalConfiguration` when the edge reports that the tunnel is not remotely managed.
- The local configuration contains only the public Quick Tunnel hostname, the loopback Gateway service, a catch-all 404 rule, and disabled warp routing. It never includes access keys, sessions, or the Dinotty origin.

## Request contract

Classification order is configuration update, WebSocket, TCP marker, control, then HTTP.
Remote configuration and TCP/WARP streams are not advertised and receive an explicit unsupported response.
Normal HTTP and WebSocket traffic always targets the loopback Gateway. `:authority` must match the
Quick Tunnel hostname and must agree with any direct or serialized `Host` header.

Cloudflare serialized headers use unpadded standard base64 `name:value` pairs separated by semicolons.
Response headers are returned through `cf-cloudflared-response-headers`, with
`cf-cloudflared-response-meta` identifying origin or connector errors. HTTP/1.1 status 101 maps to H2 status 200.

Request and response bodies stream with H2 capacity release/reservation. Content-Length is checked exactly;
unknown-length requests use HTTP/1.1 chunked framing; chunked origin responses are decoded before H2 delivery;
HEAD, 1xx, 204, and 304 have no response body. Response trailers are consumed but not forwarded in the initial implementation.
